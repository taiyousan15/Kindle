## SLIDE 03 — 普通のClaude と スキルありClaude

**タイトル:** 同じAIでも、結果がまったく違う

**内容:**
- 普通のClaude Code = **なんでも作れる家庭料理の人**
  - 美味しいものを作れる
  - でも毎回少し違う出来になる

- スキルありClaude Code = **専門分野のミシュランシェフ**
  - `/mega-research` → リサーチのミシュランシェフが担当
  - `/sdd-req100` → 航空機品質の要件定義専門家が担当
  - `/sdd-threat` → Microsoftのセキュリティアーキテクトが担当

- **同じシェフが毎回担当するから、品質がブレない**

**ビジュアル:** 左に「普通のAI」（バラバラの料理）、右に「スキルあり」（ミシュランの一皿）の対比

## SLIDE 04 — スキルの正体

**タイトル:** スキルは「専門家の手順書」を書いたファイル

**内容:**
- スキルの実体は `SKILL.md` という1枚のMarkdownファイル
- このファイルに記されているもの：
  - **どのAIモデルを使うか**（精度重視→opus、速度重視→haiku）
  - **何のツールを使っていいか**（WebSearch, Bash, Write 等）
  - **どんな手順で作業するか**（Step 1〜5 の詳細な指示書）

- Claude Code はこれを読んで **「その専門家として振る舞う」**

**ポイント（大きく表示）:**
> スキルは「優秀な人の最善手を、誰でも再現できる形にしたもの」

**ビジュアル:** SKILL.mdファイルのシンプルな構造図（モデル選択→ツール指定→手順書）

## SLIDE 06 — コマンドの解剖

**タイトル:** コマンド1行の中に、すべての指示が詰まっている

**内容（コードブロック大きく表示）:**
```
/mega-research  生成AI市場の競合分析  --mode=deep
      ↑                 ↑                  ↑
  スキル名          メインテーマ          詳細オプション
（どの専門家を呼ぶ）（何を調べるか）  （どこまで深堀りするか）
```

- 内部では `$ARGUMENTS` 変数でテーマを受け取り自動展開
- 検索クエリを **12〜18個** 自動生成
- 6つのAPIで並列検索 → 統合・重複除去・スコアリング

**ポイント:**
> テーマが変わっても手順（ノウハウ）は変わらない
> だから「生成AI市場」でも「不動産市場」でも同じ品質

**ビジュアル:** コマンド入力 → 自動展開 → レポート出力のフロー図

## SLIDE 07 — SDDにおける spec-slug

**タイトル:** 1つのIDが、12のドキュメントをまとめ上げる

**内容:**
```
/sdd-full  household-budget-app
                 ↑
           spec-slug（プロジェクトID）
```

- 同じIDを使い続けることで、すべての文書が連動する：

```
/sdd-context   household-budget-app  → business-context.md
/sdd-req100    household-budget-app  → requirements.md
/sdd-design    household-budget-app  → design.md
/sdd-threat    household-budget-app  → threats.md
                    ↓
     .kiro/specs/household-budget-app/ に一元管理
```

- 後から再実行するだけで **文書を更新・改善できる**

**ビジュアル:** 1つのIDから12のドキュメントが放射状に広がるダイアグラム

## SLIDE 15 — Groqとは何か

**タイトル:** 世界最速クラスのAI推論。しかも無料枠あり

**内容:**
- Groq = 「LPUチップ」という独自ハードウェアで動作する推論サービス

**普通のAI推論（GPUサーバー）:**
```
質問 → GPUで処理 → 2〜10秒で回答
```
**Groq（LPU）:**
```
質問 → LPUで処理 → 0.5〜1秒で回答（体感で「瞬時」）
```

**なぜリサーチに使えるか:**
- 1回あたりの処理が速い → 複数の角度から同時調査が高速
- 無料枠で個人利用は月数百〜数千回使える
- **最新の Llama 4 モデルが使える**

**ビジュアル:** スピードメーターの対比（GPU:遅い vs LPU:速い）

---

## SLIDE 16 — Llama 4 Scout の実力

**タイトル:** 軽量なのに、超長文を一気に処理できる

**内容:**
```
モデル名: llama-4-scout-17b-16e-instruct
```

**特徴:**
- **コンテキスト長: 最大524,288トークン**
  （一般的な小説1冊以上を一気に読み込める）
- **速度:** Groq LPU で超高速動作
- **構造:** 17B × 16エキスパート（MoE）— 軽量で高精度
- **コスト:** 無料枠あり・従量課金でも格安

**TAISUN v2 での設定（litellm-config.yaml）:**
```yaml
- model_name: groq-scout
  litellm_params:
    model: groq/meta-llama/llama-4-scout-17b-16e-instruct
    api_key: os.environ/GROQ_API_KEY
```

**ビジュアル:** 524,288トークン = 書籍何冊分かを可視化したグラフィック

## SLIDE 21 — Research Delegation：自動並行実行

**タイトル:** `/mega-research` と入力するだけで2つが同時に動く

**内容:**
```
あなたが /mega-research と入力
        ↓
   [自動的に2つが並行起動]
        ├── intelligence-research（バックグラウンド）
        │   31ソース: 経済指標・ニュース・X投稿...
        │
        └── mega-research（フォアグラウンド）
            6 API統合: テーマの深掘り調査...
        ↓
   [両方の結果を統合]
        ↓
   文脈 + 詳細 = 使えるレポート
```

**バックグラウンド実行の3つの理由:**
1. 時間的ロスなし（並行処理）
2. 1つのAPIが失敗しても他に影響しない
3. メインの調査のコンテキストを圧迫しない

**ビジュアル:** 2つの処理が並走する視覚的フロー図

## SLIDE 23 — 普通の要件定義書 vs TAISUN SDD

**タイトル:** 「機能リスト」と「設計の全体像」はまったく別物

**内容:**
**❌ 一般的な要件定義書（よくある問題）:**
```
機能一覧.xlsx
├── ユーザーログイン機能
├── 商品一覧表示機能
└── 決済機能
→ 3ヶ月後に「セキュリティどうする？」問題が発覚
→ 「パフォーマンス基準を決めていなかった」でリリース遅延
```

**✅ TAISUN SDD が一括生成する文書群:**
```
.kiro/specs/my-project/
├── stakeholder-map.md    誰が何を求めているか
```
