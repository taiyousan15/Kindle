## 1. そもそも「スキル」って何？

### 1-1. 「普通のClaude」と「スキルあり Claude」の違い

**普通のClaude Codeは万能だが、プロではない。**

料理で例えると、普通の Claude Code は「なんでも作れる家庭料理の人」です。美味しいものを作れますが、毎回少し違う出来になります。

スキルを追加した Claude Code は違います。

```
/mega-research   → ミシュランシェフ（リサーチ専門）が担当
/sdd-req100      → 航空機品質の要件定義専門家が担当
/sdd-threat      → Microsoftのセキュリティアーキテクトが担当
```

**同じシェフが毎回担当するから、品質がブレない。** これがスキルの本質です。

---

### 1-2. スキルの正体は「専門家の手順書」

スキルは `SKILL.md` というMarkdownファイルです。このファイルに：

- どのAIモデルを使うか（精度重視 → opus、速度重視 → haiku）
- 何のツールを使っていいか（WebSearch, Bash, Write 等）
- どんな手順で作業するか（Step 1〜5 の詳細な指示書）

が書かれています。Claude Code はこれを読んで「その専門家として振る舞う」わけです。

> 💡 **ポイント**: スキルは「AIへの指示書の標準化」です。
> 優秀な人が編み出した最善手を、誰でも再現できる形にしたもの。

---

### 1-3. 「スキルを作る側」になるとどうなるか

自分で `/competitor-lp-analysis` というスキルを作れば：

| 従来 | スキル化後 |
|-----|----------|
| 毎回「競合LPを調べてほしい」と細かく説明する | `/competitor-lp-analysis 自社サービス名` だけ |
| AI が毎回違うアプローチで調査 | 毎回同じ高品質な手順で調査される |
| チームに共有できない | SKILL.md ファイルを渡すだけで共有完了 |
| 次のプロジェクトで一から説明し直す | コマンド1つで再利用 |

**TAISUN v2 の 101スキルは、「多くの人が繰り返してきた作業」を標準化したもの。**
使うだけで、その恩恵を受けられます。

## 2. キーワードはどう機能するか

### 2-1. キーワードは「専門家への依頼内容」

```
/mega-research  生成AI市場の競合分析  --mode=deep
      ↑                 ↑                  ↑
  スキル名         メインテーマ          詳細オプション
（どの専門家を呼ぶ）  （何を調べるか）   （どこまで深堀りするか）
```

スキルの内部では `$ARGUMENTS` という変数でこれを受け取り、以下のように展開されます。

```
入力: /mega-research 生成AI市場の競合分析

スキル内部での処理:
  1. テーマ「生成AI市場の競合分析」を解析
  2. 検索クエリを自動生成（12〜18個）
     - 「生成AI 市場規模 2026 シェア」
     - 「ChatGPT Claude Gemini 競合比較 最新」
     - "generative AI competitive landscape 2026"
     - "AI startup funding Q1 2026" ...
  3. 6つのAPIで並列検索
  4. 結果を統合・重複除去・スコアリング
  5. Markdown形式でレポート生成
```

**キーワードが変わっても、手順（ノウハウ）は変わらない。**
だから「生成AI市場」でも「不動産市場」でも同じ品質で出てくるのです。

---

### 2-2. SDDにおける「spec-slug」という考え方

SDD（要件定義）スキルでは、キーワードが「プロジェクトのID」になります。

```
/sdd-full  household-budget-app
                 ↑
          これが spec-slug（仕様スラッグ）
```

このIDが**すべてのSDDスキルを繋ぎます**。

```
/sdd-context   household-budget-app  → business-context.md 生成
/sdd-req100    household-budget-app  → requirements.md 生成
/sdd-design    household-budget-app  → design.md 生成
/sdd-threat    household-budget-app  → threats.md 生成
       ↑
全部同じIDで、.kiro/specs/household-budget-app/ に蓄積される
```

**1つのIDが、12のドキュメントを一貫したプロジェクトとして管理する。**
後から「要件定義を見直したい」場合も `/sdd-req100 household-budget-app` を再実行するだけです。

### 3-2. 「無料」と「有料」で何が具体的に違うか

**例: 「最新のAIコーディングツール市場」を調査した場合**

| 情報の種類 | 無料 (/research-free) | フル (/mega-research) |
|-----------|----------------------|----------------------|
| **一般的なブログ記事** | ✅ 取得できる | ✅ 取得できる |
| **Google上位10件の情報** | △ 一部のみ | ✅ SerpAPI で完全取得 |
| **Redditの本音コメント**「実際使ってる人の感想」 | ❌ 取得不可 | ✅ Reddit API で大量収集 |
| **今日・今週のニュース**「最新のリリース情報」 | △ 不完全 | ✅ NewsAPI で2万ソース対応 |
| **AI特化の精度の高い検索** | ❌ 一般検索のみ | ✅ Tavily AI検索 |
| **英語・日本語の両方** | △ 英語が弱い | ✅ 自動で両言語検索 |

> 💡 **結論**: 無料でも「十分使える」。ただし「Redditの生の声」「今週のニュース」が必要なら有料APIが価値を発揮する。

### 3-3. world-research が「世界レベル」な理由

普通の検索は「Googleで調べる」だけです。world-research は6つの全く異なる角度から同時に見ます。

```
あなたが「生成AI代理店ビジネス」を調べたい場合

  Layer 1: SNS・コミュニティ（X, Reddit, Hacker News）
           「実際に始めた人の体験談」「失敗談」「リアルな収益」
                    ↓
  Layer 2: 学術論文・研究機関（arXiv, Google Scholar）
           「AIビジネスの市場規模推定」「将来予測の根拠」
                    ↓
  Layer 3: 注目コンテンツ（Product Hunt, GitHub Trending）
           「今まさに伸びているAIツール」「投資されているサービス」
                    ↓
  Layer 4: 実践者の知見（Medium, Substack, Zenn, Qiita）
           「経験者が語る成功の条件」「具体的なノウハウ」
                    ↓
  Layer 5: エコシステム指標（GitHub Stars, npm downloads）
           「実際に使われているか」「人気度の数値」
                    ↓
  Layer 6: ニュース・業界メディア（TechCrunch, Bloomberg）
           「公式発表」「資金調達情報」「業界動向」
```

これら6層を **同時に** 調査するから、「世間の評判」「学術的根拠」「ビジネスの現実」「開発者の実感」が一つのレポートに統合されます。

**一人の人間が同じことをやると数日かかります。** world-research は15〜20分で完了します。

## 5. Groq + Llama 4 Scout：最先端情報収集の切り札

### 5-1. Groqとは何か

**Groq（グロック）** は「LPUチップ」という独自ハードウェアで動作する、**世界最速クラスのAI推論サービス**です。

普通のAI（GPUサーバー）と比べると：

```
普通のAI推論:
  質問 → GPUで処理 → 2〜10秒で回答

Groq (LPU):
  質問 → LPUで処理 → 0.5〜1秒で回答（体感で「瞬時」）
```

**なぜリサーチに使えるのか？**
- 1回あたりの処理が速いため、「複数の角度から同時に調査」を高速にこなせる
- 無料枠が存在する（個人利用では毎月数百〜数千回無料）
- 最新の Llama 4 モデルが使える

---

### 5-2. Llama 4 Scout（llama-4-scout-17b-16e）とは

**Llama 4 Scout** は Meta が開発した最新の大規模言語モデルです。
「17B × 16エキスパート（MoE）」という構造で、**軽量ながら高精度**を実現しています。

```
モデル名: llama-4-scout-17b-16e-instruct
特徴:
  - コンテキスト長: 最大524,288トークン（非常に長い文書を一気に処理）
  - 速度: Groq LPUで超高速動作
  - 精度: 最新モデルとして情報の鮮度が高い
  - コスト: 無料枠あり・従量課金でも格安
```

TAISUN v2 では `config/litellm-config.yaml` に以下のように設定されています：

```yaml
- model_name: groq-scout
  litellm_params:
    model: groq/meta-llama/llama-4-scout-17b-16e-instruct
    api_key: os.environ/GROQ_API_KEY
    max_tokens: 8192
```

---

### 5-3. Groq + Llama 4 Scout で「最先端情報収集」ができる理由

```
他のモデルとの使い分け:

  Claude Sonnet / Opus
    → 深い推論・長文生成・構造化ドキュメント作成に強い
    → 費用: 有料（Anthropic課金）

  Groq + Llama 4 Scout
    → 超高速処理・大量の情報を素早くさばくのに強い
    → 用途: 情報の「一次スクリーニング」「速報確認」「要約作成」
    → 費用: 無料枠あり
```

**具体的にどんなリサーチに使われるか**

```
シナリオ1: 速報ニュースの一次処理
  「今週の生成AIニュース100件をGroqで高速スクリーニング
   → 重要な10件だけClaude Sonnetで深掘り」
  → 全体の処理時間を大幅短縮

シナリオ2: intelligence-research の並行処理
  「31ソースからの情報収集をGroqで高速処理
   → 経済指標・ニュース・SNS投稿を瞬時に要約」

シナリオ3: 大量テキストのリアルタイム分析
  「コンテキスト524Kトークンを活かして
   競合の長文レポートや論文を一気に読み込んで要約」
```

---

### 5-4. LiteLLM フォールバック：Groq はバックアップとしても機能

TAISUN v2 の LiteLLM 設定では、Claude Haiku が使えない時の自動代替として Groq が設定されています：

```yaml
fallbacks:
  - claude-haiku:
      - groq-scout        ← Haiku が使えない時は Groq で代替
      - openrouter-free
```

**つまり Groq は「メインモデルの代替」としても機能するため、
Groq API キーを設定するだけでシステム全体の安定性が上がります。**

## 6. intelligence-research：常時監視する情報網

### 6-1. 他のリサーチスキルとの決定的な違い

普通のリサーチスキルは「頼まれたテーマを調べる」ものです。
intelligence-research は「誰に頼まれなくても、常時31ソースを監視して情報を提供する」ものです。

| 普通のリサーチ | intelligence-research |
|--------------|----------------------|
| あなたが「〇〇を調べて」と指定する | 自動起動（Research Delegation ルール） |
| 特定テーマの詳細情報 | 今日の経済・ニュース・著名人発言・X投稿 |

**なぜ「常時監視」が必要か？**

```
❌ テーマだけ調べた場合:
   「競合他社A社のLP分析が完了しました。戦略は〜」
   ← でも今、米国金利が急上昇して競合他社は予算削減中という情報がない。
   → 的外れな戦略提案になる可能性がある

✅ intelligence-research と組み合わせた場合:
   「今週の状況: 金利〇%、AI業界でX件の資金調達、
    業界リーダーのSam Altmanが"価格競争激化"と発言。

   → 背景情報込みで意味のある分析になる
```

### 6-3. Research Delegation：自動並行実行の仕組み

`/mega-research` や `/world-research` を実行すると、**ルールで自動的に** intelligence-research も同時起動します（`research-delegation.md` に定義）。

```
あなたが /mega-research と入力
      ↓

**バックグラウンド実行にする理由:**

- メインの調査の作業メモリ（コンテキスト）を圧迫しない

## 7. 要件定義（SDD）：1コマンドで世界標準の文書が揃う

### 7-1. そもそも「要件定義」って何？（素人向け超基本）

**まず「要件定義なしで開発するとどうなるか」を想像してください。**

```
〔よくある失敗パターン〕

社長: 「家計管理アプリを作って」
開発者: 「わかりました」（2ヶ月後...）
開発者: 「できました！」
社長: 「なぜ家族で共有できないの？」
開発者: 「聞いてないです」
社長: 「セキュリティは？」
開発者: 「特に考えてないです」
社長: 「Androidでも使える？」
開発者: 「iOSしか作ってないです」

→ 2ヶ月と数百万円が水の泡
```

これが「要件定義なし」の現実です。

**要件定義とは「開発前に全員が合意する約束書」のことです。**
- 「何を作るか」
- 「誰のために作るか」
- 「どこまで作るか・作らないか」
- 「どれくらい速く・安全に動くか」

これらをテキストで文書化したものが要件定義書です。

---

### 7-2. 「要件定義書」と「SDD」は何が違うか

多くの現場でいう「要件定義書」は、こんな感じのものです：

```
❌ 一般的な要件定義書（よくある問題）

  機能一覧.xlsx
  ├── ユーザーログイン機能
  ├── 商品一覧表示機能
  └── 決済機能

  これだけで開発スタート
  → 3ヶ月後に「セキュリティどうする？」問題が発覚
  → 「パフォーマンス基準を決めていなかった」でリリース遅延
  → 「そもそも何のために作ったのか」が忘れられる
```

TAISUN SDD が生成する文書は全く違います：

```
✅ TAISUN SDD が一括生成する文書群

  .kiro/specs/my-project/
  ├── stakeholder-map.md    ← 誰が何を求めているか
  ├── business-context.md  ← なぜ作るか・KPI・競合分析
  ├── requirements.md      ← 何を作るか（EARS準拠・採点付き）
  ├── design.md            ← どう作るか（C4アーキテクチャ図）
  ├── adr-001.md           ← なぜこの技術を選んだか
  ├── slos.md              ← どの品質で動かすか（数値基準）
  ├── threats.md           ← どんな攻撃があるか（STRIDE分析）
  ├── guardrails.md        ← どう守るか
  ├── tasks.md             ← どの順番で作るか
  ├── runbook.md           ← 問題が起きたらどうするか
  ├── critique.md          ← 第三者視点の批判的レビュー
  └── score.json           ← 文書の品質スコア（100点満点）
```

**これが `/sdd-full my-project` の1コマンドで全部生成されます。**

---

### 7-3. 各スキルの「なぜ使うか」を理解する

#### sdd-context：「なぜ作るか」を最初に決める（Amazon PR-FAQ方式）

```
Amazonの製品開発の秘密:
  Amazonは新製品を作り始める前に「プレスリリース」を書きます。
  まだ製品が存在しない段階で：

    「〇〇株式会社、新機能□□を本日リリース。
     従来の〜〜という問題を解決し、顧客満足度が2倍向上」

  というプレスリリースを書き、チームで共有します。

  なぜか？ → 「このプレスリリースが書けないなら、作るべきではない」から。
```

**sdd-context が出力するもの（business-context.md）:**
- 📰 プレスリリース（Amazon PR-FAQ形式）— 製品の価値を一言で表現
- 📊 OKR — ビジネス目標と数値指標
- 🎯 KPI — 数値・単位・測定方法・達成期限が全部入り
- 🗺️ Impact Map — 誰が・どう変わる・そのために何を作るか（Mermaid図）
- 🔍 競合・代替手段分析 — 最低3つ・差別化の根拠
- ❌ Non-Goals — 「今回やらないこと + 理由」のリスト
- ⏱️ Appetite — プロジェクトの最大期間（Shape Up方式）

> **なぜ重要か**: 「要件定義」より前に「ビジネス文脈」が固まると、後から「そもそも何のために作ったのか」という混乱が起きない。

---

#### sdd-req100：EARS準拠の「曖昧ゼロ」要件定義

**EARSとは**: 英国の防衛産業（Rolls-Royce等）が、航空機エンジンのような「曖昧さが命取り」の開発のために作った要件の書き方。

```
❌ 曖昧な要件（ほとんどの現場にある）
  「システムはユーザーに通知を送信する」
  問題: いつ？どんな条件で？どこに？何を？何分以内に？

✅ EARS準拠の要件
  「ユーザーが注文確定ボタンを押した時、
   システムは注文確認メールを
   登録済みのメールアドレスに
   30秒以内に送信しなければならない」
  ✓ 全て明確 ✓ テストケースが書ける ✓ 実装者が迷わない
```

**採点方法**: 各要件を0〜10点で採点し、平均が「要件定義スコア」。
- 0〜59点: 不十分（**多くの現場がここ**）
- 60〜79点: 平均的
- 80〜89点: 高品質（大手企業レベル）
- 90〜100点: **世界標準が完全適用された状態**（SDDで達成可能）

---

#### sdd-design：C4モデルによる「誰にでも伝わる」設計図

**C4モデル** = システムを4段階の解像度で見せる設計手法。

```
Level 1: Context（コンテキスト図）
  「このシステムは何者か」
  → CEO・営業担当・非技術者向け
  例: 「ユーザーがスマホで注文 → 在庫システムに繋がる」

Level 2: Container（コンテナ図）
  「何でできているか」
  → プロジェクトマネージャー向け
  例: 「WebアプリがAPIサーバーを呼び出し、DBに保存」

Level 3: Component（コンポーネント図）
  「各部品の中に何があるか」
  → 開発チームリーダー向け

Level 4: Code（コード図）
  「クラス・シーケンスの詳細」
  → 実装担当エンジニア向け
```

> **ポイント**: C4モデルを使うと「CEOにも・開発者にも・運用担当にも」、それぞれの立場で理解できる設計書が作れます。

---

#### sdd-threat：STRIDEセキュリティ分析（Microsoft発）

**STRIDE** = Microsoftが開発した、セキュリティの「抜け漏れゼロ」確認手法。

| 文字 | 意味 | 例 |
|-----|------|-----|
| **S**poofing | なりすまし | 他人のアカウントでログインされる |
| **T**ampering | 改ざん | 通信中のデータを書き換えられる |
| **R**epudiation | 否認 | 「注文していない」と後から言われる |
| **I**nformation Disclosure | 情報漏洩 | パスワードやカード番号が流出する |
| **D**enial of Service | サービス停止 | 大量リクエストでサーバーが落ちる |
| **E**levation of Privilege | 権限昇格 | 一般ユーザーが管理者権限を取得する |

sdd-threat は全コンポーネントにSTRIDEを適用し、**リスクを数値化して優先順位を付けた文書**を生成します。

> **なぜ重要か**: セキュリティは「後から付け足す」のが最も費用がかかる。設計段階でSTRIDEを通しておくと、開発コストが大幅に下がります。

---

#### sdd-slo：「なんとなく速い」をやめる

**SLO（Service Level Objective）** = 「このシステムはどのくらい速く・安定して動くか」の数値目標。

```
❌ SLO未定義の問題（よくある）
  「できるだけ速くしてください」
  → 開発者は「速くした（比較基準なし）」
  → ユーザーは「遅い」と不満
  → 誰も正しい判断ができない

✅ SLO定義済みの場合
  可用性SLO:   99.9%（月に43.8分の停止まで許容）
  レイテンシSLO: 95%のリクエストが500ms以内
  エラー率SLO:  0.1%未満

  → 基準が明確なので「改善できたか」が数値で判断できる
  → エラーバジェット（0.1%）を超えたら機能開発より安定化優先
```

---

#### sdd-full：全13スキルを最適順序で自動実行

```
/sdd-full my-app  を実行すると...

  Phase 0: /sdd-stakeholder  → ステークホルダー分析
  Phase 1: /sdd-context      → PR-FAQ・OKR・KPI・競合分析
  Phase 2: /sdd-req100       → EARS準拠の要件定義（100点採点）
  Phase 3: /sdd-design       → C4アーキテクチャ設計
           /sdd-adr          → 技術選定の根拠記録
  Phase 4: /sdd-slo          → SLO/SLA数値定義
           /sdd-threat        → STRIDE脅威分析
           /sdd-guardrails    → セキュリティガードレール
  Phase 5: /sdd-tasks        → 実装タスク分解
           /sdd-runbook       → 運用Runbook
  Phase 6: /sdd-critique     → 批判的レビュー
           /sdd-score         → 品質スコア（100点満点）

  → 12ファイルが .kiro/specs/my-app/ に自動生成される
```

**所要時間: 数時間〜1日。従来の要件定義: 数週間〜数ヶ月。**

## 8. SDDとアジャイル：「なぜ一緒に使うのか」を素人向けに解説

### 8-1. まず「アジャイル」とは何か（超入門）

**アジャイル開発** とは「小さく作って、素早く試して、どんどん改善する」という開発の考え方です。

```
〔アジャイル以前：ウォーターフォール開発〕

  要件定義（3ヶ月）→ 設計（2ヶ月）→ 開発（6ヶ月）→ テスト（2ヶ月）→ リリース
  ↑ ここで初めて顧客が使える（合計13ヶ月後）

  問題: 13ヶ月後に「こんなもの欲しくなかった」と言われても遅い
```

```
〔アジャイル開発〕

  2週間で動くものを作る → 顧客に見せる → フィードバック → 2週間で改善
  → 2週間で改善 → 2週間で改善 → ...

  ✅ 早い段階で「方向性が合っているか」確認できる
  ✅ 途中で要望が変わっても対応できる
  ✅ 使える機能を早く届けられる
```

---

### 8-2. アジャイルだけでは「失敗する」理由

アジャイルの考え方は素晴らしいですが、**実は大きな落とし穴があります。**

```
〔アジャイルだけでよく起きる問題〕

  Sprint 1（2週間）: ログイン機能を作った
  Sprint 2（2週間）: 商品一覧を作った
  Sprint 3（2週間）: カートを作った
  Sprint 7（2週間）: 「そういえば、このサービスって何のために作ってるんだっけ？」
          ↑
          ここで「目的を忘れた開発」に気づく
```

さらに：

```
  Sprint 10（2週間）: 「セキュリティどうしますか？」← 今さら？
  Sprint 12（2週間）: 「データベースの設計が根本的に間違っている」← 全部作り直し
  Sprint 15（2週間）: 「パフォーマンスの基準って決めてましたっけ？」
```

**アジャイルは「作り方」は教えてくれますが、「何を作るか・なぜ作るか」は教えてくれません。**

---

### 8-3. SDDとアジャイルを組み合わせると何が変わるか

**SDD はアジャイルを「正しく始めるための準備」として機能します。**

```
〔SDD × アジャイルの理想的な進め方〕

  Sprint 0（1〜2日）: /sdd-full my-project を実行
  ─────────────────────────────────────────
  この時点で全員が確認済み:
    ✅ なぜ作るか（business-context.md）
    ✅ 何を作るか（requirements.md）
    ✅ どう作るか（design.md）
    ✅ セキュリティは？（threats.md）
    ✅ どのくらいの速度・品質が必要？（slos.md）
    ✅ どの順番で作るか（tasks.md）
  ─────────────────────────────────────────

  Sprint 1（2週間）: ログイン機能を作る
    → 要件が明確なので「仕様の確認」が不要
    → セキュリティ要件が最初から分かっているので後から困らない

  Sprint 2〜N（2週間ずつ）: どんどん機能を追加
    → 方向性がブレない
    → 「そもそも何のために作ってるか」を全員が共有済み
    → 途中での根本的な設計変更がほぼ起きない
```

---

### 8-4. 「Sprint 0」というやり方

アジャイルの世界には「Sprint 0」という概念があります。
最初の Sprint だけは「動くものを作る」ではなく「正しく始める準備をする」に使います。

```
従来の Sprint 0:
  - チームでWikiを整備する
  - 開発環境を構築する
  - 大まかな方向性を話し合う
  所要時間: 1〜2週間
  品質: 担当者のスキルに依存

SDD を使った Sprint 0:
  コマンド1つ: /sdd-full my-project
  所要時間: 数時間〜1日
  品質: 世界標準の手法が自動適用
  成果物: 12の文書（要件・設計・セキュリティ・品質基準すべて）
```

> 💡 **一言で言えば**: **SDDはアジャイルの「スプリント0」を、世界標準品質で超高速に実行するツールです。**
> アジャイルを「正しく・速く・高品質に」始めるための準備が整います。

---

### 8-5. SDD なしで始めた場合と比較

| タイミング | SDD あり（Sprint 0で実行） | SDD なし（アジャイルのみ） |
|----------|--------------------------|------------------------|
| **Sprint 1 前** | 要件・設計・セキュリティが確定済み | 何となく始める |
| **Sprint 3** | 仕様通りに進む | 「これで合ってたっけ？」が発生 |
| **Sprint 7** | セキュリティ対応済み | 「今からセキュリティどうする？」 |
| **Sprint 10** | 品質基準が数値で管理できている | 「なんとなく速い」で進む |
| **リリース直前** | 問題点がほぼない | 根本的な設計見直しが発覚することも |
| **リリース後** | 運用Runbookがある（runbook.md） | 問題が起きてから考える |

## 9. なぜSDDは他の手法より優れているのか

### 9-1. 従来の要件定義との比較

| 比較項目 | 従来の要件定義書 | TAISUN SDD |
|---------|--------------|-----------|
| **作成時間** | 数週間〜数ヶ月 | 数時間〜1日 |
| **品質の安定性** | 担当者のスキルに完全依存 | 世界標準の手法を毎回自動適用 |
| **セキュリティ** | 後回し・見落としがち | STRIDE分析が最初から組み込み |

| **コスト** | コンサル費用 数百万円〜 | TAISUN v2 + API代のみ |
| **更新・管理** | Wordファイルで属人化 | Markdown × Git で全員が更新可能 |
| **再利用** | 毎回ゼロから作る | 同じコマンドで同品質を再現 |

### 9-2. 「世界の天才プロセス」を1コマンドで使う

SDDが採用している手法は、すべて実績ある世界標準です。

| 手法 | 開発元 | 実績・背景 |
|-----|-------|---------|
| **Amazon PR-FAQ** | Amazon | AWS・Kindle・Primeなどすべての製品開発で使用 |
| **EARS** | Rolls-Royce / 英国防衛産業 | 航空機エンジン・軍事システムの要件定義標準 |
| **C4 Model** | Simon Brown | 世界中の大規模システム設計で採用 |
| **STRIDE** | Microsoft | Windows・Azureのセキュリティ設計 |
| **Shape Up / Circuit Breaker** | Basecamp | SaaS開発の「期間超過しない」管理手法 |
| **OKR** | Intel → Google | Google全社・多くのスタートアップで採用 |
| **Impact Mapping** | Gojko Adzic | アジャイルの「なぜ作るか」定義の標準 |

これらを個別に習得・適用しようとすると、**専門家でも数ヶ月の学習が必要**です。

**SDD は `/sdd-full` の1コマンドで、これら全てを自動的に適用します。**

### 10-2. 要件定義コストの比較

```

## 付録：よくある質問

**Q: APIキーがなくても本当に使えますか？**
はい。`/research-free` `/research-cited-report` `/note-research` はAPIキー完全不要で動作します。

**Q: まず最初に何を登録すべきですか？**
Tavily（月1,000回無料）・FRED（完全無料）・Groq（無料枠あり）の3つだけで十分スタートできます。費用はゼロ。

**Q: SDDで生成した文書は修正できますか？**
はい。全てMarkdownファイルで生成されます。手動修正後、`/sdd-score` で品質スコアを再確認できます。

**Q: アジャイルをすでにやっている会社でもSDDは使えますか？**
むしろアジャイル環境に最適です。Sprint 0（最初のスプリント）で `/sdd-full` を実行するだけで、その後のスプリントが全て効率よく進みます。

**Q: 英語と日本語どちらに対応していますか？**
両方対応しています。日本語でキーワードを入力しても、スキルが自動的に英語クエリも生成して検索します。

**Q: Ollamaは必須ですか？**
いいえ。Ollamaは「完全無料・完全オフライン」の選択肢です。Claude MAX/ProプランやOpenRouter経由でも同じスキルが使えます。

**Q: Groqのコストはどのくらいですか？**
個人利用では多くの場合、無料枠内に収まります。超過しても従量課金で月数百円〜のケースがほとんどです。

**Q: セキュリティ上の懸念はありますか？**
- Ollamaプランを使えば全処理がローカルで完結（データが外部に送信されない）
- `.env` ファイルはGitにコミットしない（`.gitignore` に追加済み）
